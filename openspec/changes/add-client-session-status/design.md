## Context

Клієнти бронюють сесії без акаунту (B2B-модель: єдиний авторизований користувач кабінету — психолог). Після бронювання клієнт бачить лише інлайн-повідомлення "Запис створено..." на сторінці `/[slug]` (`src/app/[slug]/booking-form.tsx`) і жодного способу перевірити статус пізніше. Прецедент публічної неавторизованої сторінки вже існує: `/pay/[sessionId]` (`src/app/pay/[sessionId]/page.tsx`) використовує сирий cuid сесії як URL-параметр і показує психолога, дату/час, статус оплати — без токена чи авторизації.

Окремо виявлено функціональний дефект: `cancelSession` і `rescheduleSession` (`src/app/(cabinet)/sessions/actions.ts`) оновлюють `Session` і роблять `revalidatePath("/sessions")` (сторінка психолога), але ніколи не викликають `notifyClient` — клієнт не дізнається про скасування/перенесення жодним каналом.

## Goals / Non-Goals

**Goals:**
- Клієнт може в будь-який момент перевірити поточний статус своєї сесії за лінком з сповіщення
- Клієнт отримує сповіщення (email/Telegram, за існуючою логікою `notifyClient`) при скасуванні або перенесенні сесії психологом
- Нова сторінка й нові сповіщення повторно використовують існуючі паттерни (`/pay/[sessionId]`, `notifyClient`, шаблони в `src/lib/notifications.ts`) замість нової інфраструктури

**Non-Goals:**
- Немає клієнтського акаунта, логіну чи списку/історії сесій клієнта — сторінка показує рівно одну сесію за її id
- Немає окремого публічного токена/полів у схемі — перевикористовується `Session.id`, як і в `/pay/[sessionId]`
- Без можливості для клієнта самостійно скасувати/перенести сесію через цю сторінку (лише перегляд)

## Decisions

**1. URL-ідентифікатор: сирий `Session.id` (cuid), без окремого токена**
Дзеркалить `/pay/[sessionId]`. Альтернатива (окреме поле `publicToken` з можливістю відкликання) додає міграцію, генератор токена і ще одне поле заради гіпотетичної потреби відкликати доступ, якої зараз немає. cuid2 має достатню ентропію, щоб не бути вгадуваним перебором — той самий рівень довіри вже прийнятий для сторінки оплати.

**2. Обсяг сторінки — одна сесія, без списку**
Проєкт свідомо B2B без клієнтських акаунтів (`proposal.md` кореневої зміни). Історія сесій клієнта = фактичний клієнтський кабінет із питаннями авторизації, які поза межами цієї зміни.

**3. Виправлення сповіщень при cancel/reschedule — частина цієї ж зміни, не окремий фікс**
Лінк на сторінку статусу без сповіщення про зміну статусу марний (клієнт не дізнається, що треба перейти); сповіщення без лінка — це просто повернення до поточного стану. Обидві частини вирішують один корінь проблеми, тож ідуть одним change.

**4. Формат нового тексту сповіщень (скасовано/перенесено) — за зразком існуючих `notifyClient`-викликів**
Використовує ту саму структуру каналів (email default, Telegram якщо прив'язаний, email fallback), що вже реалізована для `bookingConfirmationForClient`/`paymentStatusForClient` — нова функція просто додає новий тип контенту, не нову логіку доставки.

## Risks / Trade-offs

- **[Ризик] Сирий session id у публічному URL** технічно дозволяє будь-кому з лінком переглянути статус сесії безстроково → **Мітигація**: прийнятний ризик, ідентичний вже чинному `/pay/[sessionId]`; сторінка лише read-only і не розкриває персональних даних понад те, що клієнт сам лишив при бронюванні.
- **[Ризик] Реюз `notifyClient` для нових типів подій без явного product-рев'ю тексту** → **Мітигація**: тексти скасування/перенесення пишуться за наявним тоном інших клієнтських шаблонів у `src/lib/notifications.ts`.

## Migration Plan

Без даних для міграції (нове поле не додається). Розгортання: новий роут + оновлені шаблони + два нові виклики `notifyClient` — стандартний деплой без окремих кроків відкату, окрім звичайного revert коміту.
