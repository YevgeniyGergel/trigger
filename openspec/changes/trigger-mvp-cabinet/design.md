## Context

Trigger — новий продукт (проєкт з нуля), no existing code or specs. Ціль MVP — дати одному психологу (single-tenant per account, B2B) робочий кабінет: розклад, бронювання клієнтами без акаунту, оплата карткою, голосові нотатки сесій. Клієнти не мають власного акаунту — взаємодіють лише через публічну сторінку бронювання.

Обмеження: команда з нуля обирає стек, потрібно швидко дійти до робочого MVP; юрисдикція — Україна, тому платіжний провайдер має підтримувати гривню та карткові платежі українських банків.

## Goals / Non-Goals

**Goals:**
- Психолог керує розкладом, клієнтами, сесіями та оплатами в одному кабінеті
- Клієнт бронює консультацію за публічним посиланням без реєстрації
- Оплата карткою прив'язана до конкретної сесії, статус синхронізований через webhook
- Психолог може надиктувати нотатку після сесії, отримати транскрипт і прив'язати до клієнта
- Архітектура дозволяє додати multi-tenant (кілька психологів на одному домені) пізніше без перепису ядра

**Non-Goals:**
- Мультитенантний маркетплейс психологів для пошуку клієнтами (пізніший change)
- Відеодзвінки/телемедицина в кабінеті (інтеграція з Zoom/Google Meet — окремий change)
- Розсилки, email-маркетинг, нагадування SMS (окремий change)
- Мобільний застосунок (лише responsive web в MVP)
- Складна тарифікація/підписка самого SaaS для психолога (окремий change, спочатку — ручний онбординг)

## Decisions

**Стек: Next.js (App Router, TypeScript) + PostgreSQL + Prisma + NextAuth**
- Один репозиторій для UI кабінету, публічної сторінки бронювання та API routes — менше операційної складності для MVP одного розробника.
- Prisma дає типобезпечні міграції та зручний доступ до Postgres, зручно для реляційних даних (клієнти/сесії/оплати).
- NextAuth (email+пароль або magic link) достатньо для одного типу користувача (психолог); клієнти взагалі не автентифікуються.
- Альтернатива: окремий backend (NestJS) + SPA — відхилено, бо додає операційну складність без потреби на MVP-масштабі.

**Платежі: LiqPay, мерчант-акаунт на кожного психолога**
- Кожен психолог реєструє власний мерчант-акаунт у LiqPay самостійно (Trigger не може зробити це програмно) і вставляє видані `public_key`/`private_key` у налаштуваннях кабінету. Ключі зберігаються з application-level шифруванням (KMS/secrets manager), не просто encryption-at-rest — щоб доступ до розшифрованого значення мав лише сервісний процес застосунку, а не пряме читання з БД.
- Обрано LiqPay замість Monobank Acquiring: **Monobank вимагає, щоб розрахунковий рахунок ФОП був відкритий саме в Monobank** — це жорстке обмеження, яке відсіювало б психологів з рахунком в інших банках (Приват, Райф тощо). LiqPay такої прив'язки до конкретного банку психолога не має.
- Кожен платіж передає `rro_info` (email для фіскального чека) — LiqPay має власний вбудований безкоштовний ПРРО ("Каса") для підключених компаній, окрема інтеграція з Checkbox чи іншим ПРРО не потрібна.
- Модель оплати: психолог позначає сесію "платна", генерується LiqPay checkout-посилання; клієнт оплачує на публічній сторінці підтвердження; статус оновлюється через LiqPay webhook (server callback), а не лише client-side redirect. Невдала спроба оплати переводить статус у "failed" (клієнт може повторити — новий checkout повертає статус у "pending").
- Режим підтвердження бронювання — налаштування психолога: **ручне підтвердження** (default) або **обов'язкова передоплата** (доступно лише з активною LiqPay-інтеграцією). У режимі передоплати слот тримається конфігурований період; успішна оплата авто-підтверджує сесію, неоплачена сесія після дедлайну авто-скасовується (cron), слот звільняється, клієнт отримує сповіщення. Сесії без ціни завжди йдуть через ручне підтвердження.
- Повернення коштів: при скасуванні оплаченої сесії кабінет пропонує refund через LiqPay API (ключі мерчанта психолога вже є); успіх → paymentStatus "refunded" + сповіщення клієнту; психолог може відмовитись від refund згідно своєї політики скасування.
- Відкрите питання на майбутнє: додати Monobank Acquiring як другий провайдер на вибір психолога (кращий бренд-довіра для тих, у кого рахунок вже в Monobank) — вимагає абстракції "провайдер оплати" в моделі `Payment`, не блокує MVP.

**Сповіщення: Email + Telegram**
- Email — базовий канал через транзакційний email-провайдер (напр. Resend/SendGrid): підтвердження бронювання, нагадування за N годин до сесії, підтвердження оплати.
- Telegram — через власного Telegram-бота Trigger: клієнт один раз прив'язує свій акаунт до запису (перехід за посиланням з deep-link `t.me/TriggerBot?start=<token>` під час бронювання), далі бот надсилає ті самі типи сповіщень. Бот працює лише "на вихід" (notifications), без діалогової логіки бронювання в MVP — це відкладено як можливий наступний change.
- Психолог отримує сповіщення про нове бронювання/скасування/оплату тими ж каналами, які налаштував у профілі.

**Голосові нотатки: браузерний запис (MediaRecorder API) → завантаження аудіо → асинхронна транскрипція → SOAP-структурування (killer feature)**
- Аудіо зберігається в об'єктному сховищі (S3-сумісне, напр. Cloudflare R2), транскрипція — окрема фонова job (виклик Whisper API), результат записується в БД і прив'язується до сесії.
- Після успішної транскрипції другий фоновий крок викликає Claude API, щоб структурувати сирий транскрипт у формат SOAP (Subjective/Objective/Assessment/Plan) — це основний диференціатор Trigger від generic booking-CRM (EasyWeek, Altegio), які не мають клінічної документації взагалі.
- Транскрипція та SOAP-структурування асинхронні (job queue / polling статусу), щоб не блокувати UI на довгих аудіо.
- Психолог може відредагувати і сирий транскрипт, і структурований SOAP-текст вручну (STT та LLM не ідеальні для спеціалізованої лексики, редагування — обов'язкова частина флоу, не опція).

**Шифрування нотаток: application-level, ключ у KMS**
- Явний компроміс: психолог хоче, щоб адмін БД не мав прямого доступу до нотаток, але сервер повинен розшифрувати текст, щоб відправити його в Claude API для SOAP-структурування — це виключає справжній zero-knowledge (клієнт-side) підхід.
- Обране рішення: `transcriptText`/`editedText` шифруються на рівні застосунku ключем, що зберігається в KMS/secrets manager, доступним лише сервісній ролі бекенду. Це захищає від витоку/дампа БД (адмін БД, що має лише доступ до Postgres, бачить лише шифротекст), але не від скомпрометованого прод-сервера — прийнятний рівень для vertical SaaS такого масштабу (аналогічно працює SimplePractice).
- Аудіофайли в об'єктному сховищі — encryption at rest на рівні бакета, доступ лише через підписані URL з обмеженим TTL.

**Модель даних (спрощено):**
- `Psychologist` (auth-обліковка, профіль, публічний slug для бронювання)
- `Client` (належить психологу, не має логіну)
- `AvailabilitySlot` / `WorkingHours` (розклад психолога)
- `Session` (booking): client, psychologist, timeslot, status (pending/confirmed/cancelled/completed), paymentStatus
- `Payment`: sessionId, provider=liqpay, amount, status, providerTransactionId
- `SessionNote`: sessionId, audioUrl, transcriptText, status (recording/transcribing/ready), editedText

**Публічне бронювання без акаунту клієнта**
- Захист від зловживань (спам-бронювання): rate-limiting за IP/телефоном на сторінці бронювання, підтвердження бронювання не обов'язкове через email/SMS у MVP (спрощення), але психолог бачить і може скасувати підозрілі записи.

**Часовий пояс: `TZ=Europe/Kyiv` на рівні процесу**
- Розклад/слоти/блокування побудовані на "локальному часі сервера" (`Date.getHours()`, `setHours()` тощо) — найпростіший підхід для одноюрисдикційного MVP, але він **ламається**, якщо сервер виконується не в українському часовому поясі. Vercel serverless-функції за замовчуванням працюють у UTC (Україна — UTC+2/+3), тож без явної фіксації весь розклад психолога зсунувся б на 2-3 години в проді.
- Рішення: змінна середовища `TZ=Europe/Kyiv` встановлюється явно для dev/build/test через `cross-env` у `package.json` та обов'язково додається як env var у Vercel production settings (задача в tasks.md секція 10). Це змушує всю "локальну" арифметику дат Node.js коректно означати київський час незалежно від regionu хостингу.
- Альтернатива (переписати всю логіку на date-fns-tz з явною конвертацією wall-clock↔UTC) — розглядалась, але відкладена як надлишкова складність для MVP з однією юрисдикцією; варта повернення, якщо продукт вийде за межі України.
- Клієнтський бік: серверна арифметика лишається в Europe/Kyiv, але **публічні клієнтські сторінки (бронювання, статус сесії) відображають час у часовому поясі браузера відвідувача** з явною позначкою пояса, якщо він відрізняється від київського. IANA-пояс клієнта фіксується при бронюванні, зберігається на записі клієнта і використовується в клієнтських сповіщеннях (сегмент клієнтів-українців за кордоном).

**CI/CD: GitHub Actions + Vercel preview/production deploys**
- Репозиторій — git з нуля, хостинг на GitHub. GitHub Actions на кожен pull request: lint, typecheck, unit/integration tests, перевірка Prisma-міграцій на конфлікти.
- Vercel підключений напряму до репозиторію: кожен PR отримує preview-деплой автоматично; злиття в `main` тригерить production-деплой через той самий пайплайн (не окремий ручний крок).
- Branch protection на `main` — вимагає проходження CI перед merge, щоб не зламати production код, що вже приймає платежі клієнтів.
- Міграції БД застосовуються як частина deploy step (не вручну), щоб уникнути розсинхронізації схеми між середовищами.
- Моніторинг з першого дня прод-запуску: базовий error tracking (напр. Sentry) та алерти на збої LiqPay/Telegram webhook і фонових job (транскрипція, SOAP-структурування) — це процеси, що падають мовчки, якщо не спостерігати за ними явно.

## Risks / Trade-offs

- [Webhook LiqPay недоступний/затримується] → статус оплати може відставати від реальності → показувати "очікує підтвердження" в UI, реалізувати ручну звірку/повторний запит статусу психологом
- [Транскрипція голосу помиляється на психологічній термінології] → нотатки неточні → завжди дозволяти ручне редагування тексту, не покладатись на STT як on-the-record джерело
- [Публічне бронювання без автентифікації клієнта] → спам/фейкові записи → rate limiting + можливість психолога вручну відхилити/заблокувати номер
- [Зберігання чутливих аудіо/тексту нотаток сесій психотерапії] → високі вимоги до приватності → шифрування at rest в об'єктному сховищі, обмежений доступ лише власнику-психологу, видалення на запит
- [Одна велика зміна (весь MVP одним change)] → більший ризик регресій і довший цикл до першого релізу → tasks.md структурувати по вертикальних зрізах (спочатку auth+клієнти, потім бронювання, потім оплата, потім нотатки, потім сповіщення), щоб можна було тестувати інкрементально навіть у межах одного change
- [SOAP-структурування через Claude API помиляється або галюцинує клінічний контент] → недостовірні нотатки, довіра психолога падає → завжди позначати SOAP-текст як чернетку до ручного підтвердження психологом, ніколи не показувати як фінальний документ без явного review
- [Telegram-бот недоступний/заблокований у психолога чи клієнта] → сповіщення не доходять → email лишається обов'язковим fallback-каналом незалежно від того, чи прив'язаний Telegram

## Migration Plan

Проєкт з нуля — міграції як такої немає. Розгортання: Vercel (застосунок) + managed Postgres (напр. Neon/Railway) + Cloudflare R2 (аудіо). Rollback стратегія на MVP-етапі — редеплой попереднього білда через Vercel.

**Регіон хостингу: EU (Франкфурт/Амстердам)**
- База даних містить чутливі дані (нотатки психотерапії, контакти клієнтів) — розміщення в EU дає найближчу до України географію (нижча латентність) і GDPR-рівень захисту даних, що доречно для медичних/психотерапевтичних записів навіть попри те, що Trigger — не EU-компанія.
- Postgres (Neon/Railway) — обрати EU-регіон (напр. `eu-central-1`/Frankfurt) при провіжининні.
- Vercel — deployment regions налаштувати на EU (Frankfurt `fra1`), щоб serverless-функції виконувались близько до БД і не додавали крос-регіональну латентність.
- Cloudflare R2 — бакет в EU jurisdiction (Cloudflare підтримує EU data residency для R2).
- KMS/secrets manager для шифрування нотаток — обрати провайдера з EU-регіоном (напр. AWS KMS `eu-central-1` або Vercel/Doppler з EU storage); конкретний провайдер — відкрите питання, вирішити на етапі 1.6 tasks.md.
